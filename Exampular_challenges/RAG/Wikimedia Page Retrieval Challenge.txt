Wikimedia Multi Page Retrieval Dataset

Overview

This dataset was constructed from the English Wikipedia database dump (enwiki-latest-pages-articles.xml.bz2), provided by the Wikimedia Foundation under the Creative Commons Attribution-ShareAlike 4.0 (CC BY-SA 4.0) license. The raw Wikipedia XML was parsed to extract article text, which was then split by section headings to produce individual documents. Each document was assigned an opaque identifier (doc_id) — no article titles, section names, or source URLs are exposed in the public dataset.

Given a natural-language research question, the task is to retrieve an ordered set of 2–5 document IDs from this corpus that together provide the information needed to fully answer the question.

In real-world information retrieval, answers are rarely found on a single page — and the pages you need aren't always discoverable from the question alone. A student researching degree requirements might start with the course catalog, discover a prerequisite they didn't know about, then look up that prerequisite's syllabus, and finally check the credit transfer policy. Each step reveals what to look for next.

This dataset models that challenge. The corpus contains ~318,000 documents spanning ~50,000 topics across history, science, geography, philosophy, biography, politics, music, technology, and more. The 1,500 research questions require consulting 2–5 documents each, with many questions requiring chain-of-retrieval — reading initial results to discover which additional documents are needed.

bridged: 500 questions (33.3%) — Multiple documents within a single topic are connected through shared context not explicitly stated in the question.



parallel: 437 questions (29.1%) — All required documents can be retrieved independently from the question text alone.

String escaping: String fields may contain commas, quotes, and special characters, properly escaped per RFC 4180



Text content: The text column contains cleaned plaintext with all wiki markup removed

No missing values: Neither file contains null or missing values in any column

Wikimedia Page Retrieval
Overview
When humans research a complex topic, they open multiple pages, follow links from one to another, and piece together the answer across sources.

This challenge asks solvers to do the same: given a research question, retrieve the ordered set of 2–5 documents from a large opaque corpus that together answer it. Many questions require chain-of-retrieval — reading an initial document to discover which subsequent documents are needed. The corpus provides only opaque doc_ids and plaintext — no titles, categories, or structural metadata.

Questions follow three implicit retrieval patterns (labels are not provided per question):

Sequential (~38%): Document A must be read to discover Document B exists.
Bridged (~33%): Documents are connected through context not stated in the question.
Parallel (~29%): All documents can be found independently from the question.
Evaluation
Chain-Aware Precision-Weighted F-score (F-0.5) with an order bonus.

Set accuracy uses F-0.5 (precision weighted 2x over recall):

F-0.5 = 1.25 × P × R / (0.25 × P + R)
Each question has 2–5 ground truth documents. Submitting fewer means lower recall; submitting more tanks precision. The grader does not reject submissions outside this range — it scores them, and F-0.5 handles the penalty naturally.

Order bonus (chain questions only): Kendall tau over correctly retrieved documents, contributing up to ±20%:

score = F-0.5 × (1 + 0.25 × tau) / 1.25     # sequential/bridged
score = F-0.5                                  # parallel
Final score = mean across all 1,200 test questions. Higher is better.

Draft
The solution must return valid document IDs that exist in the provided corpus (`corpus.csv`) for every question

Submissions with fabricated or hallucinated document IDs that don't match any entry in the corpus will score zero on those questions. The agent must treat the corpus as the closed universe of possible answers.

DATA HANDLING
REQUIRED
The solution must be selective — returning too many documents is penalized.

The scoring uses precision-weighted F-0.5, which penalizes over-retrieval. An agent that returns 10+ documents hoping to include the correct 3 will score lower than one that precisely returns just the correct 3. Quality over quantity.

DATA HANDLING
REQUIRED
The agent should implement iterative retrieval — reading initially retrieved documents to discover what additional documents are needed

Many questions are designed so that the question text alone does not contain enough information to find all relevant documents. The agent should retrieve an initial set, analyze their content, and perform follow-up retrieval based on what it learned. Single-pass retrieval will underperform on chain questions.

AGENT BEHAVIOR
RECOMMENDED
The solution should build a retrieval index (embeddings, inverted index, or similar) rather than performing brute-force comparison across all 350K documents.

The corpus contains ~350,000 documents. Naively comparing every question against every document without indexing is computationally expensive. Strong solutions use vector embeddings, BM25, or hybrid retrieval structures for efficient candidate generation

AGENT BEHAVIOR
RECOMMENDED
The solution should predict document IDs in the correct consultation order, not just as an unordered set.

The scoring includes an ordering component (Kendall tau). A solution that finds all correct documents but returns them in reversed order will score up to 20% lower than one with correct ordering. The agent should model retrieval order as part of its prediction.
